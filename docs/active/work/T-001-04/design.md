# T-001-04 Design: Dockerfile

## Approach: Phoenix 1.8 Multi-Stage Dockerfile

### Option A: Two-stage (build + runtime)
Standard Phoenix pattern. Single build stage handles deps, compilation, assets, and release. Runtime stage copies the release.

**Pros:** Simple, well-documented, fewer layers to manage.
**Cons:** No cache optimization between dep fetch and compile.

### Option B: Three-stage (deps + build + runtime)
Separate deps stage caches `mix deps.get` + `mix deps.compile` separately from app compilation.

**Pros:** Better layer caching — deps don't recompile on app code changes.
**Cons:** Slightly more complex, but standard in Phoenix community.

### Option C: Four-stage (deps + assets + build + runtime)
Further splits asset building into its own stage.

**Pros:** Maximum cache granularity.
**Cons:** Over-engineered for this project — assets are simple (no npm).

### Decision: Option B (three-stage)

Three stages: `deps`, `build`, `runtime`.

Rationale:
- Deps compilation is the slowest step (Ash ecosystem is large). Caching it separately from app code is worth the minor complexity.
- Assets don't need a separate stage since there's no npm/node — `mix assets.deploy` runs inline during the build stage.
- This matches the pattern `mix phx.gen.release --docker` generates for Phoenix 1.8.

### Base Images

**Build stage:** `hexpm/elixir:1.19.0-erlang-28.0-debian-bookworm-20250317-slim`
- Official hex.pm image with exact version pins
- Debian bookworm-slim keeps build image reasonable
- Includes Elixir, Erlang, Mix, Hex, Rebar

**Runtime stage:** `debian:bookworm-20250317-slim`
- Same Debian version as build (library compatibility)
- No Elixir/Erlang — release embeds ERTS
- ~80MB base, release adds ~20-30MB → well under 100MB target

### Version Pinning Strategy
- Elixir and OTP versions as Dockerfile ARGs with defaults matching `mise.toml`
- Debian version pinned in both stages for reproducibility
- Comment referencing `mise.toml` and `ci.yml` for version sync

### Release Configuration
- Use `mix release` defaults (embedded ERTS, no custom vm.args needed yet)
- Generate `rel/overlays/bin/migrate` via `mix phx.gen.release` for migrations
- Create custom `rel/overlays/bin/migrate_and_start` that runs migrations then starts

### migrate_and_start Script
```sh
#!/bin/sh
bin/migrate
exec bin/server
```
- `bin/migrate` runs Ecto migrations (generated by phx.gen.release)
- `bin/server` starts with PHX_SERVER=true (generated by phx.gen.release)
- `exec` replaces shell process so signals reach the BEAM

### .dockerignore
Essential to keep build context small and avoid leaking secrets:
- `_build/`, `deps/`, `.git/`, `docs/`, `test/`, `.github/`
- Dev/editor files, env files

### Rejected Alternatives
- **Alpine:** Smaller images but Erlang NIF compatibility issues (musl vs glibc). Not worth the debugging risk.
- **Distroless:** No shell available, makes debugging harder. Minimal benefit over slim.
- **Multi-arch build:** Not needed — Fly.io runs amd64. Can add later if needed.
