# T-003-03 Structure: Photo Upload

## Files Created

### `lib/haul/storage.ex`
Storage abstraction module.
- `put_object(key, binary, content_type)` — dispatches to backend
- `delete_object(key)` — dispatches to backend
- `public_url(key)` — returns URL for the stored object
- `upload_key(tenant, prefix, filename)` — generates S3 key
- Private functions for `:local` and `:s3` backends

### `lib/haul/storage/local.ex`
Local filesystem backend for dev/test.
- Writes to `priv/uploads/` directory
- Returns file path for `public_url`

### `lib/haul/storage/s3.ex`
S3 backend for production (Fly Tigris).
- Uses `ExAws.S3` for put/delete operations
- Generates public URLs from bucket + key

### `test/haul/storage_test.exs`
Unit tests for storage module (local backend).

### `test/haul_web/live/booking_live_upload_test.exs`
Separate test file for upload-specific booking tests.
- Photo input renders with correct attributes
- File selection shows previews
- Successful upload flow
- Form submittable without photos
- Max file count enforcement
- Invalid file type rejection

## Data Flow

```
User selects photos → LiveView validates (type, size, count)
                     → Thumbnails shown via live_img_preview
                     → User can cancel individual uploads

User submits form → handle_event("submit")
                  → consume_uploaded_entries → Haul.Storage.put_object per file
                  → Collect S3 keys into list
                  → Merge photo_urls into form params
                  → AshPhoenix.Form.submit()
                  → If storage fails: submit without photos (graceful fallback)
```
